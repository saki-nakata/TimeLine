# TimeLine シーケンス図

**バージョン:** 1.0
**作成日:** 2026-05-24
**作成者:** Nakata Saki

---

## 概要

本ドキュメントでは、フロントエンド・バックエンド・DB 間の時系列インタラクションを定義する。
認証フロー（ログイン・自動リフレッシュ・ログアウト）とプロフィール編集（アバターアップロード）を対象とする。

### トークン仕様（前提知識）

| 種別 | 生成方式 | 有効期限 | Cookie 設定 |
|------|---------|---------|------------|
| アクセストークン | HMAC-SHA256 署名付き JWT（sub = userId） | 15 分 | HttpOnly, path=`/`, SameSite=Strict |
| リフレッシュトークン | 乱数 32 byte → Base64URL。DB には SHA-256 ハッシュのみ保存（ローテーション方式） | 7 日 | HttpOnly, path=`/api/auth`, SameSite=Strict |

`refresh_token` Cookie は path=`/api/auth` のため、`/api/auth/**` へのリクエスト時のみブラウザが自動送信する。

## 1. ユーザー登録

```mermaid
sequenceDiagram
    participant FE as フロントエンド
    participant BE as バックエンド
    participant DB as DB

    FE->>BE: POST /api/auth/register<br/>{ username, email, password }

    BE->>DB: SELECT * FROM users WHERE email = ?
    DB-->>BE: null（未登録）
    BE->>DB: SELECT * FROM users WHERE username = ?
    DB-->>BE: null（未登録）

    BE->>BE: bcrypt.encode(password)
    BE->>DB: INSERT INTO users<br/>(username, email, password_hash)
    DB-->>BE: 登録完了（id 採番）

    BE->>BE: JWT 生成（sub=userId, 有効期限 15 分）
    BE->>BE: リフレッシュトークン生成<br/>（乱数 32 byte → Base64URL）
    BE->>DB: INSERT INTO refresh_tokens<br/>(user_id, SHA-256(token), expires_at)
    DB-->>BE: 保存完了

    BE-->>FE: 201 Created + UserResponse<br/>Set-Cookie: access_token（HttpOnly, path=/, 15 分）<br/>Set-Cookie: refresh_token（HttpOnly, path=/api/auth, 7 日）

    FE->>FE: currentUser にセット → /home へ遷移
```

## 2. ログイン

```mermaid
sequenceDiagram
    participant FE as フロントエンド
    participant BE as バックエンド
    participant DB as DB

    FE->>BE: POST /api/auth/login<br/>{ email, password }

    BE->>DB: SELECT * FROM users WHERE email = ?
    DB-->>BE: User レコード

    BE->>BE: bcrypt.matches(password, password_hash)

    alt パスワード不一致 または ユーザー不存在
        BE-->>FE: 401 Unauthorized
    else 認証成功
        BE->>BE: JWT 生成（sub=userId, 有効期限 15 分）
        BE->>BE: リフレッシュトークン生成
        BE->>DB: INSERT INTO refresh_tokens<br/>(user_id, SHA-256(token), expires_at)
        DB-->>BE: 保存完了

        BE-->>FE: 200 OK + UserResponse<br/>Set-Cookie: access_token（HttpOnly, path=/, 15 分）<br/>Set-Cookie: refresh_token（HttpOnly, path=/api/auth, 7 日）

        FE->>FE: currentUser にセット → /home へ遷移
    end
```

## 3. 認証済み API リクエスト（通常時）

アクセストークンが有効な場合のリクエスト処理フロー。

```mermaid
sequenceDiagram
    participant FE as フロントエンド
    participant Filter as JwtAuthFilter
    participant BE as バックエンド
    participant DB as DB

    Note over FE: ブラウザが access_token Cookie を自動送信

    FE->>Filter: GET /api/posts<br/>Cookie: access_token=<JWT>

    Filter->>Filter: Cookie から access_token を取り出す
    Filter->>Filter: JWT 署名検証 + 有効期限確認
    Filter->>Filter: sub（userId）を取り出し<br/>SecurityContext にセット

    Filter->>BE: リクエスト続行

    BE->>DB: SELECT posts ...（クエリ実行）
    DB-->>BE: 結果

    BE-->>FE: 200 OK + レスポンスデータ
```

## 4. アクセストークン自動リフレッシュ

アクセストークンが期限切れの場合、Axios インターセプターが自動的にリフレッシュして元のリクエストをリトライする。

```mermaid
sequenceDiagram
    participant FE as フロントエンド<br/>（Axios インターセプター）
    participant BE as バックエンド
    participant DB as DB

    FE->>BE: GET /api/posts<br/>Cookie: access_token=<期限切れ JWT>

    BE->>BE: JwtAuthFilter: JWT 検証 → 期限切れ
    Note over BE: SecurityContext に認証情報がセットされない

    BE-->>FE: 401 Unauthorized

    Note over FE: interceptors.response で 401 を検知<br/>_retry フラグをチェック（無限ループ防止）

    FE->>BE: POST /api/auth/refresh<br/>Cookie: refresh_token=<raw token><br/>（path=/api/auth なので自動送信）

    BE->>BE: SHA-256(raw_token)
    BE->>DB: SELECT * FROM refresh_tokens<br/>WHERE token_hash = ?
    DB-->>BE: RefreshToken レコード

    BE->>BE: 有効期限チェック

    alt トークン無効または期限切れ
        BE-->>FE: 401 Unauthorized
        FE->>FE: window.location.href = '/login'
    else トークン有効
        BE->>DB: DELETE FROM refresh_tokens<br/>WHERE token_hash = ?（使用済みトークン削除）
        DB-->>BE: 削除完了

        BE->>BE: 新しい JWT 生成（15 分）
        BE->>BE: 新しいリフレッシュトークン生成
        BE->>DB: INSERT INTO refresh_tokens<br/>（新トークンのハッシュ）
        DB-->>BE: 保存完了

        BE-->>FE: 200 OK + UserResponse<br/>Set-Cookie: access_token（新規, 15 分）<br/>Set-Cookie: refresh_token（新規, 7 日）

        Note over FE: 元のリクエストをリトライ
        FE->>BE: GET /api/posts<br/>Cookie: access_token=<新 JWT>
        BE-->>FE: 200 OK + レスポンスデータ
    end
```

## 5. ログアウト

```mermaid
sequenceDiagram
    participant FE as フロントエンド
    participant BE as バックエンド
    participant DB as DB

    FE->>BE: POST /api/auth/logout<br/>Cookie: refresh_token=<raw token>

    BE->>BE: Cookie から refresh_token を取り出す
    BE->>BE: SHA-256(raw_token)
    BE->>DB: DELETE FROM refresh_tokens<br/>WHERE token_hash = ?
    DB-->>BE: 削除完了

    BE->>BE: access_token Cookie をクリア（maxAge=0）
    BE->>BE: refresh_token Cookie をクリア（maxAge=0）

    BE-->>FE: 200 OK<br/>Set-Cookie: access_token（maxAge=0）<br/>Set-Cookie: refresh_token（maxAge=0）

    FE->>FE: currentUser = null
    FE->>FE: /login へ遷移
```

## 6. プロフィール編集（アバターアップロード含む）

プロフィール情報の更新とアバター画像の変更は独立した 2 つのリクエストで行う。

### 6-1. プロフィール情報更新（displayName / bio / username）

```mermaid
sequenceDiagram
    participant FE as フロントエンド
    participant BE as バックエンド
    participant DB as DB

    FE->>BE: PUT /api/users/{id}<br/>{ displayName, bio, username }<br/>Cookie: access_token=<JWT>

    BE->>BE: JwtAuthFilter: JWT 検証 → userId 取得
    BE->>BE: userId == pathVariable {id} の確認（本人チェック）

    alt username 変更あり
        BE->>DB: SELECT * FROM users WHERE username = ?
        DB-->>BE: null（重複なし）または既存レコード

        alt username 重複
            BE-->>FE: 409 Conflict
        end
    end

    BE->>DB: UPDATE users SET<br/>display_name, bio, username, updated_at<br/>WHERE id = ?
    DB-->>BE: 更新完了

    BE->>DB: SELECT users + followed_by_current_user<br/>（UserProfileResponse 用）
    DB-->>BE: 最新プロフィールデータ

    BE-->>FE: 200 OK + UserProfileResponse

    FE->>FE: プロフィール表示を更新
```

### 6-2. アバター画像アップロード

```mermaid
sequenceDiagram
    participant FE as フロントエンド
    participant BE as バックエンド
    participant FS as ファイルシステム<br/>（開発: ローカル / 本番: S3）
    participant DB as DB

    FE->>BE: POST /api/users/{id}/avatar<br/>multipart/form-data: file=<画像><br/>Cookie: access_token=<JWT>

    BE->>BE: JwtAuthFilter: JWT 検証 → userId 取得
    BE->>BE: MIME タイプ検証（JPEG / PNG / WebP / GIF）
    BE->>BE: ファイルサイズ検証（最大 5MB）

    alt バリデーション失敗
        BE-->>FE: 400 Bad Request
    else バリデーション通過
        BE->>BE: UUID ファイル名生成<br/>（{uuid}.{ext}）
        BE->>FS: uploads/avatars/{uuid}.{ext} に保存
        FS-->>BE: 保存完了

        BE->>DB: UPDATE users SET<br/>avatar_url = '/uploads/avatars/{uuid}.{ext}'<br/>WHERE id = ?
        DB-->>BE: 更新完了

        BE->>DB: SELECT users + followed_by_current_user
        DB-->>BE: 最新プロフィールデータ

        BE-->>FE: 200 OK + UserProfileResponse

        FE->>FE: アバター画像を更新
    end
```
