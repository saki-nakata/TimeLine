## 1. インフラ概要

本番環境は AWS 上に構築する。EC2 を使用せず、フロントエンドは S3 + CloudFront、バックエンドは ECS Fargate で実行する構成とする。

**設計方針：**
- EC2 を使用しない（OS 管理・パッチ適用の運用コストを削減）
- NAT Gateway を使用しない（ECS をパブリックサブネットに配置しコスト削減）
- Terraform でインフラをコード管理し、再現性を確保する

## 3. AWS リソース構成

### 3.1 ネットワーク（VPC）

| リソース | 説明 |
|---------|------|
| VPC | 本番環境専用の仮想ネットワーク（10.0.0.0/16） |
| インターネットゲートウェイ | VPC をインターネットに接続 |
| パブリックサブネット（ap-northeast-1a） | ALB・ECS Fargate を配置 |
| パブリックサブネット（ap-northeast-1c） | ALB・ECS Fargate を配置（マルチ AZ） |
| プライベートサブネット（ap-northeast-1a） | RDS を配置 |
| プライベートサブネット（ap-northeast-1c） | RDS を配置（マルチ AZ） |

> **NAT Gateway は使用しない。** ECS Fargate をパブリックサブネットに配置し Public IP を付与することで ECR へのアクセスを確保。セキュリティグループで ALB からのアクセスのみ許可する。

### 3.2 セキュリティグループ

| 名前 | インバウンド | アウトバウンド | 用途 |
|------|------------|--------------|------|
| ALB 用 SG | 80・443（インターネットから） | すべて許可 | ALB への外部アクセス制御 |
| ECS 用 SG | 8080（ALB SG から） | すべて許可 | ECS タスクへのアクセスを ALB に限定 |
| RDS 用 SG | 5432（ECS SG から） | すべて許可 | DB アクセスを ECS に限定 |

### 3.3 CloudFront

| 項目 | 内容 |
|------|------|
| オリジン1 | S3（フロントエンド静的ファイル）+ OAC |
| オリジン2 | ALB（バックエンド API） |
| `/*` キャッシュビヘイビア | S3 へ転送（キャッシュあり）、SPA ルーティング関数適用 |
| `/api/*` キャッシュビヘイビア | ALB へ転送（キャッシュなし） |
| WebSocket | CloudFront が自動処理（`Upgrade`・`Connection` ヘッダーの明示的転送は不要・不可） |
| HTTPS | CloudFront デフォルト証明書（*.cloudfront.net） |
| SPA フォールバック | 403・404 → index.html（React Router 対応） |

### 3.4 S3（フロントエンド）

| 項目 | 内容 |
|------|------|
| 用途 | React ビルド成果物（dist/）の配信 |
| パブリックアクセス | すべてブロック（CloudFront OAC 経由のみ） |
| バケットポリシー | CloudFront サービスプリンシパルからの GetObject のみ許可 |

### 3.5 ALB（Application Load Balancer）

| 項目 | 内容 |
|------|------|
| タイプ | Application Load Balancer |
| 配置 | パブリックサブネット（マルチ AZ） |
| リスナー | HTTP（80）→ ECS ターゲットグループへ転送 |
| ターゲットグループ | ECS タスク（ポート 8080）、IP タイプ |
| ヘルスチェック | GET `/actuator/health` |
| Stickiness | lb_cookie（86400 秒）有効 ← WebSocket 対応 |

### 3.6 ECS Fargate

| 項目 | 内容 |
|------|------|
| クラスター起動タイプ | FARGATE |
| vCPU | 0.5 |
| メモリ | 1GB |
| タスク数 | 1（学習用） |
| 配置 | パブリックサブネット（Public IP 付与） |
| タスク実行ロール | AmazonECSTaskExecutionRolePolicy + SSM アクセス |
| タスクロール | S3 画像バケットへの読み書き権限 |
| ログ | CloudWatch Logs（/ecs/timeline、保持 30 日） |

### 3.7 ECR

| 項目 | 内容 |
|------|------|
| リポジトリ名 | timeline-backend |
| イメージスキャン | push 時に自動スキャン |
| ライフサイクル | 最新 5 件のみ保持（古いイメージを自動削除） |

### 3.8 RDS

| 項目 | 内容 |
|------|------|
| インスタンスクラス | db.t3.micro |
| エンジン | PostgreSQL 16 |
| ストレージ | 20GB（gp2） |
| 配置 | プライベートサブネット（2 AZ） |
| 外部アクセス | 不可（publicly_accessible = false） |
| 自動バックアップ | 7 日間保持 |

### 3.9 S3（画像ストレージ）

| 項目 | 内容 |
|------|------|
| バケット名 | saki-timeline-images（既存） |
| アクセス方法 | ECS タスクロール経由（アクセスキー不使用） |
| 対応フォーマット | JPEG / PNG / GIF |

### 3.10 SSM Parameter Store

| パラメータ名 | 用途 |
|------------|------|
| /timeline/db_password | RDS パスワード（SecureString） |
| /timeline/jwt_secret | JWT シークレットキー（SecureString） |

## 4. コスト概算（東京リージョン・月額）

| サービス | 月額概算 |
|---------|---------|
| CloudFront（小規模） | ~$1 以下 |
| S3 フロントエンド | ~$0.1 以下 |
| S3 画像ストレージ | ~$0.5〜2 |
| ALB | **~$18** |
| ECS Fargate（vCPU 0.5・1GB） | **~$15** |
| RDS（db.t3.micro） | **~$15**（無料枠期間中は $0） |
| ECR | ~$0〜1 |
| ACM / SSM | $0 |
| **合計** | **約 $50/月（約 7,500 円/月）** |

> **節約のポイント**: 使わないときは ECS タスク数を 0、RDS を停止することで ~$30/月 削減できる。ALB は止められない（常時 ~$18/月）。

## 5. Terraform ディレクトリ構成

```
infra/
  ├─ main.tf               # プロバイダー設定
  ├─ variables.tf          # 変数定義
  ├─ outputs.tf            # 出力値（CloudFront URL・ECR URI など）
  ├─ vpc.tf                # VPC・サブネット・セキュリティグループ
  ├─ s3.tf                 # S3（フロントエンド用）+ OAC
  ├─ cloudfront.tf         # CloudFront ディストリビューション
  ├─ ecr.tf                # ECR リポジトリ
  ├─ alb.tf                # ALB・リスナー・ターゲットグループ
  ├─ ecs.tf                # ECS クラスター・タスク定義・サービス・IAM・SSM
  ├─ rds.tf                # RDS インスタンス・サブネットグループ
  ├─ terraform.tfvars.example  # 変数値のサンプル（要コピー）
  └─ terraform.tfvars      # 実際の変数値（.gitignore 対象）
```

## 6. デプロイ手順

### 6.1 初回インフラ構築

```bash
# 1. 変数ファイルを作成
cp infra/terraform.tfvars.example infra/terraform.tfvars
# terraform.tfvars を編集して db_password・jwt_secret を設定

# 2. Terraform を実行
cd infra
terraform init
terraform plan
terraform apply

# 3. 出力値を確認
terraform output
```

### 6.2 アプリのデプロイ（コード変更時）

```bash
# フロントエンド＋バックエンド両方
bash scripts/deploy.sh

# フロントエンドのみ
bash scripts/deploy.sh frontend

# バックエンドのみ
bash scripts/deploy.sh backend
```

### 6.3 ローカルに必要なもの

- Terraform CLI
- AWS CLI（`~/.aws/credentials` に認証情報を設定）
- Docker Desktop

## 7. ローカル開発環境

本番 AWS 環境とは別に、Docker Compose を使用してローカル開発環境を構築する。

```
開発者のブラウザ
     │
     ├── localhost:5173 → Vite（React 開発サーバー）
     │
     └── localhost:8080 → Spring Boot（ローカル起動）
                              ↓
                         localhost:5432 → PostgreSQL（Docker）
```

| 環境 | フロントエンド | バックエンド | DB |
|------|------------|------------|-----|
| ローカル | Vite dev server (5173) | Spring Boot (8080) | Docker PostgreSQL |
| 本番 (AWS) | S3 + CloudFront | ECS Fargate (8080) | RDS PostgreSQL |

## 8. セキュリティ方針

| 項目 | 対応 |
|------|------|
| DB パスワード | SSM Parameter Store（SecureString）で管理 |
| JWT シークレット | SSM Parameter Store（SecureString）で管理 |
| RDS アクセス | ECS セキュリティグループからのみ許可（インターネット非公開） |
| S3 フロントエンド | CloudFront OAC 経由のみアクセス可能（パブリックアクセスブロック） |
| S3 画像 | ECS タスクロール経由（アクセスキーを使用しない） |
| ECS タスク | ALB セキュリティグループからのポート 8080 のみ許可 |
| SSH | EC2 を使用しないため不要 |
