# 認証機能定義書

**バージョン:** 1.1
**作成日:** 2026-05-18
**更新日:** 2026-05-20
**作成者:** Nakata Saki

---

## 1. 機能概要

ユーザーがアプリを利用するための認証・認可機能。メールアドレスとパスワードによるユーザー登録・ログイン・ログアウトを提供する。認証には JWT（JSON Web Token）を使用し、HttpOnly Cookie に保存することで XSS 攻撃からトークンを保護する。アクセストークン（15分）とリフレッシュトークン（7日）の2トークン方式を採用し、有効期限切れを自動でリカバリする。

---

## 2. ユーザーストーリー

| ID | ストーリー |
|----|---------|
| US-AUTH-01 | 未登録ユーザーとして、ユーザー名・メールアドレス・パスワードを入力してアカウントを作成したい |
| US-AUTH-02 | 登録済みユーザーとして、メールアドレスとパスワードを入力してログインしたい |
| US-AUTH-03 | ログイン済みユーザーとして、ログアウトボタンを押してセッションを終了したい |
| US-AUTH-04 | ログイン済みユーザーとして、ブラウザをリロードしてもログイン状態が維持されてほしい |

---

## 3. 機能詳細

### 3.1 ユーザー登録（F-AUTH-01）

#### 入力項目

| フィールド | 型 | 必須 | 制約 |
|----------|----|------|------|
| username | 文字列 | 必須 | 1〜50 文字、半角英数字・アンダースコアのみ（`^[a-zA-Z0-9_]+$`） |
| email | 文字列 | 必須 | メール形式 |
| password | 文字列 | 必須 | 8 文字以上 |

#### 正常系フロー

1. クライアントが `POST /api/auth/register` にリクエストを送信
2. サーバーがバリデーションを実行
3. username・email の重複がないことを DB で確認
4. パスワードを BCrypt でハッシュ化
5. users テーブルにレコードを INSERT
6. アクセストークン（15分）・リフレッシュトークン（7日）を生成し、それぞれ HttpOnly Cookie にセット
7. 登録されたユーザー情報（password_hash を除く）を 201 Created で返却

#### 異常系フロー

| 条件 | レスポンス |
|------|----------|
| バリデーションエラー | 400 Bad Request |
| username が既に使用されている | 409 Conflict |
| email が既に使用されている | 409 Conflict |

---

### 3.2 ログイン（F-AUTH-02）

#### 入力項目

| フィールド | 型 | 必須 | 制約 |
|----------|----|------|------|
| email | 文字列 | 必須 | — |
| password | 文字列 | 必須 | — |

#### 正常系フロー

1. クライアントが `POST /api/auth/login` にリクエストを送信
2. email でユーザーを DB から検索
3. BCrypt でパスワードを照合
4. アクセストークン（15分）・リフレッシュトークン（7日）を生成し、それぞれ HttpOnly Cookie にセット
5. ユーザー情報（password_hash を除く）を 200 OK で返却

#### 異常系フロー

| 条件 | レスポンス |
|------|----------|
| email が存在しない | 401 Unauthorized |
| パスワードが一致しない | 401 Unauthorized |

---

### 3.3 ログアウト（F-AUTH-03）

#### 正常系フロー

1. クライアントが `POST /api/auth/logout` にリクエストを送信
2. サーバーが `access_token` / `refresh_token` Cookie を Max-Age=0 で上書き（即時失効）
3. DB のリフレッシュトークンレコードを削除
4. 204 No Content を返却

認証なしで呼び出し可能（ログイン中でなくてもエラーにしない）。

---

### 3.4 現在のユーザー取得（F-AUTH-04）

#### 正常系フロー

1. クライアントが `GET /api/auth/me` にリクエストを送信（Cookie 自動付与）
2. サーバーが Cookie の JWT を検証し、ユーザー ID を取得
3. DB からユーザー情報を取得し 200 OK で返却

#### 異常系フロー

| 条件 | レスポンス |
|------|----------|
| Cookie がない／JWT が無効 | 401 Unauthorized |
| JWT 有効期限切れ | 401 Unauthorized |

フロントエンドはこのエンドポイントをアプリ起動時に呼び出し、ログイン状態を復元する。

---

### 3.5 トークンリフレッシュ（F-AUTH-05）

#### 正常系フロー

1. API リクエストに対して 401 が返ってくる
2. フロントエンドの axios インターセプターが `POST /api/auth/refresh` を自動で呼び出す
3. サーバーが `refresh_token` Cookie を検証し、DB のハッシュと照合
4. 検証成功時：旧リフレッシュトークンを DB から削除し、新しいアクセストークン・リフレッシュトークンを再発行（ローテーション）
5. フロントエンドが元のリクエストを自動リトライ

#### 異常系フロー

| 条件 | レスポンス |
|------|----------|
| `refresh_token` Cookie がない | 401 Unauthorized |
| リフレッシュトークンが DB に存在しない | 401 Unauthorized |
| リフレッシュトークンの有効期限切れ | 401 Unauthorized |

リフレッシュも失敗した場合、フロントエンドはログイン画面へリダイレクトする。

---

## 4. 使用 API エンドポイント

| Method | エンドポイント | 認証 | 説明 |
|--------|--------------|------|------|
| POST | `/api/auth/register` | 不要 | ユーザー登録 |
| POST | `/api/auth/login` | 不要 | ログイン |
| POST | `/api/auth/logout` | 不要 | ログアウト |
| POST | `/api/auth/refresh` | 不要（refresh_token Cookie） | トークン再発行 |
| GET | `/api/auth/me` | 必要 | 現在のユーザー情報取得 |

## 5. バリデーション仕様

| フィールド | ルール | エラーメッセージ |
|----------|----|------|
| username | 必須 | ユーザー名は必須です |
| username | 1〜50 文字 | ユーザー名は 1〜50 文字で入力してください |
| username | `^[a-zA-Z0-9_]+$` | ユーザー名は半角英数字・アンダースコアのみ使用できます |
| email | 必須 | メールアドレスは必須です |
| email | メール形式 | 有効なメールアドレスを入力してください |
| password | 必須 | パスワードは必須です |
| password | 8 文字以上 | パスワードは 8 文字以上で入力してください |

## 6. セキュリティ仕様

### アクセストークン（JWT）

| 項目 | 内容 |
|------|------|
| アルゴリズム | HS256 |
| 有効期限 | 15 分 |
| シークレット | 環境変数 `JWT_SECRET` で管理 |
| クレーム | `sub`（ユーザー ID）、`iat`（発行日時）、`exp`（有効期限） |

### リフレッシュトークン

| 項目 | 内容 |
|------|------|
| 生成方式 | SecureRandom 32バイト → Base64URL エンコード |
| 保存方式 | SHA-256 ハッシュ化して DB（refresh_tokens テーブル）に保存 |
| 有効期限 | 7 日 |
| ローテーション | 使用のたびに旧トークンを削除し新トークンを再発行 |

### Cookie

| 名前 | HttpOnly | Secure | SameSite | Max-Age |
|------|----------|--------|----------|---------|
| `access_token` | true | false（開発）/ true（本番） | Strict | 900（15分） |
| `refresh_token` | true | false（開発）/ true（本番） | Strict | 604800（7日） |

### パスワード

- BCrypt（strength=10）でハッシュ化して保存
- 平文パスワードは保存・ログに出力しない

### CSRF

SameSite=Strict Cookie により、クロスオリジンからのリクエストで Cookie が送信されないため、追加の CSRF トークンは不要。Spring Security の CSRF 保護は無効化する。

## 7. 制約・前提条件

| 分類 | 内容 |
|------|------|
| 認証方式 | メール + パスワード認証のみ（OAuth は将来対応） |
| トークン方式 | アクセストークン（15分）＋ リフレッシュトークン（7日）の2トークン方式 |
| セッション管理 | サーバー側でセッションを保持しない（ステートレス） |
| HTTPS | 学習目的のため現時点では HTTP のみ。本番では HTTPS 必須 |

---

## 8. 対象外仕様

| 対象外機能 | 理由 |
|-----------|------|
| OAuth（Google / GitHub 等）ログイン | スコープ外（将来対応） |
| パスワードリセット（メール送信） | スコープ外 |
| メールアドレス確認 | スコープ外 |
| 多要素認証（MFA） | スコープ外 |
