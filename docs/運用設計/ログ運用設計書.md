# ログ運用設計書

## 1. 概要

本書は TimeLine バックエンド（Spring Boot）におけるログの仕様・監視方針・保存方針・監視サービス連携方法を定義する。

---

## 2. ログ仕様

### 2.1 出力フォーマット

| 環境 | フォーマット | プロファイル |
|------|------------|------------|
| ローカル開発 | テキスト（可読形式） | `local` |
| ステージング / 本番 | JSON（1 行 1 イベント） | `staging` / `prod` |

**JSON ログの全フィールド定義（全ログ行に共通）:**

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `timestamp` | string (ISO-8601 UTC) | ログ出力時刻 |
| `level` | string | `INFO` / `WARN` / `ERROR` / `DEBUG` |
| `logger` | string | 出力元クラス名（短縮形） |
| `message` | string | 人間が読めるログメッセージ |
| `traceId` | string | リクエスト追跡 ID（UUID 32 桁、ハイフンなし） |
| `userId` | string | 認証ユーザー ID、未認証時は `"anonymous"` |
| `app` | string | 固定値 `"timeline"` |
| `env` | string | Spring Profile 名（`local` / `prod` など） |

**例外発生時のみ追加されるフィールド:**

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `stack_trace` | string | スタックトレース（JSON フィールドとして分離） |

**業務イベントログの追加フィールド例:**

| フィールド | 用途例 |
|-----------|-------|
| `event` | `"login_success"` / `"post_created"` など、イベントの機械識別名 |
| `statusCode` | HTTP ステータスコード（アクセスログ・エラーハンドラー） |
| `method` | HTTP メソッド（アクセスログ） |
| `path` | リクエストパス（アクセスログ） |
| `durationMs` | リクエスト処理時間（ミリ秒） |
| `postId` | 対象投稿 ID |
| `followerId` / `targetId` | フォロー操作の当事者 ID |
| `sizeBytes` | S3 アップロードファイルサイズ |

### 2.2 ログレベル定義

| レベル | 用途 | 例 |
|-------|------|----|
| ERROR | 即時確認が必要な障害。5xx エラー、外部サービス障害 | S3 アップロード失敗、予期しない例外 |
| WARN | 劣化状態または業務上の注意が必要な事象 | ログイン失敗、JWT 検証エラー、S3 削除失敗 |
| INFO | 正常な業務イベント | ログイン成功、投稿作成、HTTP アクセスログ |
| DEBUG | 開発・調査時のみ。本番では出力しない | JWT 期限切れ、JWT 無効トークン |

### 2.3 業務イベント一覧

| イベント (`event` フィールド) | レベル | 出力箇所 |
|---------------------------|-------|---------|
| `user_registered` | INFO | AuthService |
| `register_conflict` | INFO | AuthService |
| `login_success` | INFO | AuthService |
| `login_failed` | WARN | AuthService |
| `token_refreshed` | INFO | AuthService |
| `refresh_token_invalid` | WARN | AuthService |
| `logout` | INFO | AuthService |
| `post_created` | INFO | PostService |
| `post_updated` | INFO | PostService |
| `post_deleted` | INFO | PostService |
| `follow` | INFO | FollowService |
| `unfollow` | INFO | FollowService |
| `avatar_updated` | INFO | UserService |
| `s3_upload` | INFO | S3StorageService |
| `s3_delete` | INFO | S3StorageService |
| `s3_upload_failed` | ERROR | S3StorageService |
| `s3_delete_failed` | WARN | S3StorageService |
| `jwt_expired` | DEBUG | JwtAuthFilter |
| `jwt_invalid` | DEBUG | JwtAuthFilter |

### 2.4 ログ出力禁止事項

以下の情報はいかなるログにも含めてはならない。

- パスワード・パスワードハッシュ
- JWT トークン文字列（access token / refresh token）
- リフレッシュトークンのハッシュ値
- メールアドレスの値（フィールド名としての記録は可）
- Cookie の値

## 3. 監視項目と閾値

4つのゴールデンシグナル（Latency / Traffic / Errors / Saturation）をこのアプリに適用した監視指標を定義する。

### 3.1 Latency（レイテンシ）

| 指標 | 計測方法 | 目標値 | 要確認値 |
|-----|---------|-------|---------|
| HTTP リクエスト p95 応答時間 | アクセスログの `durationMs` | < 500ms | > 1,000ms |
| HTTP リクエスト p99 応答時間 | アクセスログの `durationMs` | < 2,000ms | > 5,000ms |
| ログイン処理時間 | `event=login_success` の `durationMs` | < 500ms | > 1,500ms |

### 3.2 Traffic（トラフィック）

| 指標 | 計測方法 | 説明 |
|-----|---------|------|
| リクエスト数 / 分 | アクセスログのカウント | 急増は攻撃またはバズを示す可能性がある |
| ログイン試行数 / 分 | `event=login_failed` のカウント | 10回/分を超えたら総当たり攻撃を疑う |
| 投稿作成数 / 時間 | `event=post_created` のカウント | スパム検知の基準として利用 |

### 3.3 Errors（エラー率）

| 指標 | 計測方法 | 目標値 | 要確認値 |
|-----|---------|-------|---------|
| 5xx エラー率 | アクセスログの `statusCode >= 500` / 総リクエスト数 | < 0.1% | > 1% |
| 4xx エラー率 | アクセスログの `statusCode >= 400` / 総リクエスト数 | < 5% | > 20% |
| S3 アップロード失敗率 | `event=s3_upload_failed` / `event=s3_upload` | < 0.1% | > 1% |
| リフレッシュトークン無効率 | `event=refresh_token_invalid` / 総リフレッシュ試行 | < 1% | > 5% |

### 3.4 Saturation（飽和度）

| 指標 | 確認箇所 | 目標値 | 要確認値 |
|-----|---------|-------|---------|
| JVM ヒープ使用率 | JVM メトリクス（Actuator / Micrometer） | < 70% | > 85% |
| スレッドプール使用率 | Tomcat スレッド数 | < 70% | > 90% |
| DB 接続プール使用率 | HikariCP メトリクス | < 80% | > 90% |
| DB スロークエリ | PostgreSQL slow query log (> 1,000ms) | 0件 / 時間 | 1件以上 / 時間 |

## 4. ログ保存・ローテーション方針

| 環境 | 保存先 | 保存期間 | 圧縮・ローテーション |
|------|-------|---------|------------------|
| ローカル開発 | コンソール（ファイル不要） | - | - |
| ステージング | CloudWatch Logs / ELK | 30 日 | 30 日経過後に自動削除 |
| 本番 | CloudWatch Logs / ELK | 90 日（監査ログは 1 年） | 圧縮アーカイブ後に削除 |

**監査ログ対象イベント（長期保存）:**
- `login_success` / `login_failed`
- `user_registered`
- `follow` / `unfollow`
- 5xx エラーログ全件

## 5. 監視サービス連携方法

本アプリのログは stdout（標準出力）に JSON 形式で出力するため、エージェントを標準出力に向けるだけで連携可能。

### 5.1 Datadog

```yaml
# docker-compose や ECS タスク定義に追加
environment:
  - DD_LOGS_ENABLED=true
  - DD_LOGS_INJECTION=true   # traceId を APM と自動相関
  - DD_SERVICE=timeline
  - DD_ENV=prod
```

Datadog Agent が stdout を自動収集する。ログパイプラインで `source:java` を設定すると `stack_trace` が自動パースされる。

**CloudWatch Logs Insights クエリ例:**
```
fields timestamp, level, traceId, userId, event, message
| filter event = "login_failed"
| stats count() by bin(5m)
```

### 5.2 AWS CloudWatch Logs

ECS / EC2 では CloudWatch Logs ドライバーを設定する。

```json
"logConfiguration": {
  "logDriver": "awslogs",
  "options": {
    "awslogs-group": "/timeline/backend",
    "awslogs-region": "ap-northeast-1",
    "awslogs-stream-prefix": "app"
  }
}
```

**Logs Insights クエリ例（5xx エラー集計）:**
```
fields @timestamp, traceId, userId, statusCode, message
| filter statusCode >= 500
| stats count() by bin(1m)
| sort @timestamp desc
```

### 5.3 ELK スタック（Elasticsearch + Logstash + Kibana）

Filebeat を stdout に向けて JSON コーデックを指定する。

```yaml
# filebeat.yml
filebeat.inputs:
  - type: container
    paths:
      - /var/lib/docker/containers/*/*.log
    json.keys_under_root: true
    json.add_error_key: true
```

`traceId` フィールドで Kibana の検索・絞り込みが可能。

### 5.4 traceId によるリクエスト追跡

障害調査の起点として `traceId` を使用する。

1. レスポンスの `X-Trace-Id` ヘッダーから traceId を取得
2. 監視サービスで `traceId = "xxx"` で絞り込む
3. 対象リクエストの全ログが時系列で表示される
