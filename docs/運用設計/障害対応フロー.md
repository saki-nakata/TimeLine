# 障害対応フロー

## 1. 概要

本書は TimeLine バックエンドにおける障害の検知・トリアージ・調査・復旧・振り返りの手順を定義する。
ログの調査は `traceId` を起点に行い、監視サービス（Datadog / CloudWatch Logs Insights 等）を使用する。

---

## 2. 障害の優先度分類（トリアージ基準）

| 優先度 | 条件 | 対応目標時間 | 対応者 |
|-------|------|------------|-------|
| **P1** 緊急 | サービス全停止 / 5xx エラー率 > 5% 継続 / ログイン不可 | 即時（15 分以内に着手） | オンコール担当 |
| **P2** 高 | 特定機能が使用不可 / 5xx エラー率 > 1% / S3 アップロード全滅 | 1 時間以内に着手 | 担当チーム |
| **P3** 中 | 応答が遅い（p95 > 1s） / 一部 API で 4xx 増加 | 次の営業日中 | 担当チーム |

---

## 3. 障害対応フロー

```
[検知]
  ↓
[一次確認・トリアージ] (P1 / P2 / P3 の分類)
  ↓
[原因調査] (ログ・メトリクス確認)
  ↓
[暫定対処] (トラフィック遮断・ロールバックなど)
  ↓
[恒久対処] (コード修正・設定変更・デプロイ)
  ↓
[復旧確認]
  ↓
[振り返り（ポストモーテム）]
```

---

## 4. 各フェーズの手順

### 4.1 検知フェーズ

障害は以下の経路で検知される。

| 検知経路 | 内容 |
|---------|------|
| 監視サービスの通知 | エラー率・レイテンシ閾値超過 |
| ユーザーからの報告 | 「ログインできない」「投稿できない」などの問い合わせ |
| ヘルスチェック失敗 | `/actuator/health` への死活監視 |

### 4.2 一次確認（5 分以内）

まず全体状況をすばやく把握する。

**手順:**

1. **HTTP アクセスログで全体状況を確認する**

   ```
   # CloudWatch Logs Insights
   fields timestamp, method, path, statusCode, durationMs, userId, traceId
   | filter statusCode >= 500
   | sort timestamp desc
   | limit 20
   ```

2. **エラー率の時系列を確認する**

   ```
   # 5xx エラー率の推移
   fields timestamp, statusCode
   | filter statusCode >= 500
   | stats count() as errorCount by bin(1m)
   | sort @timestamp desc
   ```

3. **特定のエンドポイントに集中していないか確認する**

   ```
   fields path, statusCode
   | filter statusCode >= 500
   | stats count() as cnt by path
   | sort cnt desc
   ```

4. P1 / P2 / P3 を判定し、必要に応じてエスカレーションする

### 4.3 原因調査

`traceId` を起点に個別リクエストを深掘りする。

**手順:**

1. **エラーが発生した `traceId` を特定する**

   ```
   fields timestamp, traceId, message, statusCode
   | filter statusCode >= 500
   | sort timestamp desc
   | limit 10
   ```

2. **その `traceId` でログ全体を時系列で確認する**

   ```
   fields timestamp, level, logger, event, message, stack_trace
   | filter traceId = "対象のtraceId"
   | sort timestamp asc
   ```

3. **スタックトレースを確認する**

   ```
   fields timestamp, message, stack_trace
   | filter traceId = "対象のtraceId" and level = "ERROR"
   ```

4. **認証系の問題かどうかを確認する（ログイン失敗急増など）**

   ```
   fields timestamp, event, userId
   | filter event in ["login_failed", "refresh_token_invalid", "jwt_invalid"]
   | stats count() as cnt by bin(1m)
   | sort @timestamp desc
   ```

5. **S3 の問題かどうかを確認する**

   ```
   fields timestamp, event, key, message
   | filter event in ["s3_upload_failed", "s3_delete_failed"]
   | sort timestamp desc
   ```

#### 主な障害パターンと調査の起点

| 症状 | 調査の起点 |
|------|----------|
| ログイン不可 | `event=login_failed` の急増 / DB 接続エラーの有無 |
| 投稿作成エラー | `event=s3_upload_failed` / DB エラー / `statusCode=500` の `path=/api/posts` |
| 画像が表示されない | `event=s3_upload` / `event=s3_delete` の有無・S3 の IAM ポリシー |
| 全 API が遅い | `durationMs` の分布 / DB スロークエリ / JVM ヒープ |
| 特定ユーザーのみエラー | `userId` で絞り込み → その `traceId` を追跡 |
| JWT エラーが急増 | `event=jwt_expired` / `event=jwt_invalid` のカウント |

### 4.4 暫定対処

**原因が特定できた場合:**

| 原因 | 暫定対処 |
|------|---------|
| コードのバグ | 前バージョンへのロールバック |
| DB 接続過多 | 接続プールサイズの一時拡大 / 読み取り専用クエリを Read Replica に振り分け |
| S3 障害 | 画像アップロード機能の一時無効化（機能フラグで制御） |
| メモリリーク | アプリケーションの再起動 |
| ブルートフォース攻撃 | 対象 IP / ユーザーのレート制限・ブロック |

**原因が不明の場合:**

- ロールバックを検討する（直近デプロイ後に問題が発生している場合）
- ステータスページを更新し、ユーザーへ状況を通知する

### 4.5 復旧確認

1. **エラー率が正常範囲に戻ったことを確認する**（5xx < 0.1%）
2. **レイテンシが正常範囲に戻ったことを確認する**（p95 < 500ms）
3. **主要業務フローを手動で確認する**
   - ユーザー登録 → ログイン → 投稿作成 → フォロー

---

## 5. 振り返り（ポストモーテム）

P1 / P2 障害は復旧後 2 営業日以内にポストモーテムを実施する。

### 5.1 ポストモーテムの構成

| 項目 | 内容 |
|------|------|
| 障害概要 | 発生日時・継続時間・影響範囲 |
| タイムライン | 検知 → 対応 → 復旧の時系列 |
| 根本原因 | なぜ発生したか（技術的な根因） |
| 影響 | 影響ユーザー数・リクエスト数・ビジネスへの影響 |
| 対応内容 | 暫定対処・恒久対処の内容 |
| 再発防止策 | TODO リスト（担当者・期限付き） |

### 5.2 再発防止策の例

| 根本原因 | 再発防止策 |
|---------|----------|
| DB 接続プールの枯渇 | 接続プールサイズの調整・スロークエリの最適化 |
| 環境変数の設定漏れ | デプロイチェックリストへの追加 |
| 外部依存（S3）の障害 | S3 障害時の fallback 処理の実装 |
| メモリリーク | コードレビューの強化・定期的な負荷テスト |
